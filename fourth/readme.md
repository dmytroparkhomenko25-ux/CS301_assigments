# 1 - 9 рядки (create_tables.sql)
Дропаємо таблиці, щоб уникнути конфліктів при перестворенні бази даних та очистити застарілі дані перед запуском схеми.

# 11 - 76 рядки
Створюємо 6 таблиць для роботи таксі: riders, drivers, vehicles, rides, payments та driver_status_logs. Тут налаштовано зв'язки 1:1, 1:багато та багато-до-багатьох, а також додано CHECK-обмеження на правильність координат, рейтингів від 1 до 5, статусів та сум. Також у кінці створюємо композитний індекс на статус та час зміни для прискорення аналітики.

# 1 - 32 рядки
Створюємо процедуру request_and_assign_ride, яка перевіряє наявність пасажира та активність водія в системі, створює нову поїздку та тимчасово робить водія неактивним, оскільки він уже на замовленні.

# 34 - end
Створюємо тригер trg_log_driver_status_change та функцію fn_log_driver_status_change, яка автоматично логує кожну зміну статусу активності водія в таблицю driver_status_logs для подальшого аудиту.

# create_view.sql
Створюємо представлення driver_performance_summary, яке об'єднує дані водія, автомобіля, кількість завершених замовлень та загальний заробіток для бізнес-звітів.

# create_user.sql
Оголошуємо 3 ролі з різними правами доступу: ride_admin буде мати повний доступ, ride_analyst - тільки читання таблиць та представлень, та ride_service_app аля читання/запис транзакційних даних, але без права на зміну структури DDL.
